# Codex 开发汇报与验收规范

本规范用于约束 Codex 每次开发任务结束后的汇报格式，目标是让非技术产品负责人能判断实现效果，也让 ChatGPT 能基于证据做审查。

## 核心原则

每次任务只围绕一个明确目标或一条用户路径汇报。不要只说明“改了什么”，必须说明“验收项是否通过”和“证据是什么”。

状态只允许使用：`通过` / `失败` / `未验证` / `不适用`。

没有实际运行验证的内容不得写成 `通过`，必须写 `未验证` 并说明原因。

## 最终汇报格式

每次开发任务结束后，必须按以下格式输出：

```md
## 本轮开发汇报

### A. 本轮目标
- 用户路径：
- 验收标准：
- 范围限制：

### B. 修改摘要
- 修改文件：
- 核心变化：
- 刻意未做：

### C. 验收结果
| 验收项 | 状态 | 证据 |
| --- | --- | --- |
| 项目启动/构建 | 通过/失败/未验证/不适用 | 命令 + 关键输出 |
| API 行为 | 通过/失败/未验证/不适用 | 请求 + 响应摘要 |
| 前端用户路径 | 通过/失败/未验证/不适用 | 截图/浏览器验证/Network 记录 |
| 测试/Smoke | 通过/失败/未验证/不适用 | 命令 + 关键输出 |

### D. 阻塞点
- 阻塞步骤：
- 证据：
- 影响：

### E. 风险与假设
- 风险：
- 假设：

### F. 下一步
- 只建议一个最小动作：
```

## 证据要求

优先提供以下证据：

1. 页面截图或浏览器自动化结果。
2. API 请求和响应摘要。
3. 终端命令结果。
4. 测试或 smoke 输出。
5. 相关文件路径和关键代码位置。

## 禁止写法

不要使用以下表达作为完成依据：

- “已修复”
- “应该可以”
- “理论上可运行”
- “逻辑已完善”
- “建议用户自己测试”

如果没有证据，只能写 `未验证`。

## 下一步规则

如果本轮没有完全通过，最终结论必须写 `部分通过` 或 `失败`。

下一步只能建议一个最小动作，不能一次性提出多个开发方向。

推荐格式：

```md
最终结论：完全通过 / 部分通过 / 失败
下一步只建议：<一个最小动作>
需要用户判断的问题：<仅限产品选择，不问技术实现选择>
```

## ChatGPT 审查友好要求

为了让 ChatGPT 能快速审查，Codex 的最终汇报必须包含：

- 本轮目标是否达成。
- 哪些验收项通过。
- 哪些验收项失败或未验证。
- 每个结论对应的证据。
- 是否有范围外改动。
- 下一步只做什么。

不要把大段过程日志当作验收报告。过程日志只在能证明问题或结果时引用关键片段。
